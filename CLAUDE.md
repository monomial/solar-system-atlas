# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

"Helios ‚Äî Solar System and Galactic Atlas": an interactive 3D map built with Three.js on a Next.js App Router surface. It is a **fully client-side static site** ‚Äî no server, no API, no database. Four scales, one "you are here" chain: the solar-system layer renders heliocentric positions for any date between 1800-01-01 and 2050-12-31; the Milky Way, Local Group, and Observable Universe layers place our system in progressively larger cosmic contexts, each scale's home marker positioned so it lands exactly where the previous scale's whole view sits.

Deployed to GitHub Pages at https://monomial.github.io/solar-system-atlas/ by `.github/workflows/deploy.yml` on every push to `main`.

`tvos/HeliosTV/` is a SwiftUI/SceneKit sibling app ‚Äî an ambient, narrated tvOS screensaver that tours the same catalog and mirrors the same finale (Milky Way ‚Üí Local Group ‚Üí cosmic web) in Swift/Metal. It shares data with the web build one-way: `scripts/emit-catalog.mjs` generates `tvos/HeliosTV/Media/bodies.json` from `app/bodies.ts` and copies textures, narration clips, and the universe field/CMB assets into the tvOS bundle. The web app never reads anything from `tvos/`. See `tvos/README.md` for the Swift-side architecture; this file covers the web build.

It was originally generated with ChatGPT and hosted on ChatGPT Sites (a Cloudflare Workers/vinext stack, with a server route that proxied NASA/JPL Horizons). That was all removed when it moved to Pages; if you find references to vinext, wrangler, D1, or `/api/dwarf-positions`, they are stale.

## Commands

- `npm run dev` ‚Äî dev server on :3000.
- `npm run build` ‚Äî regenerates the catalog (see below), then static export to `out/`.
- `npm run catalog` ‚Äî runs `scripts/emit-catalog.mjs` (writes `tvos/HeliosTV/Media/bodies.json`, copies textures/narration/universe assets into the tvOS bundle) and `scripts/emit-orbit-goldens.mjs` (regenerates the tvOS Swift test fixtures). Runs automatically as part of `build`; run it by hand after touching `app/bodies.ts`, `app/cosmic.ts`, or any committed texture/narration asset if you need the tvOS bundle current without a full build.
- `npm run preview` ‚Äî serve the built export.
- `npm test` ‚Äî builds, then runs `tests/` (export integrity, orbital math, cosmic catalogs, the universe field).
- `npm run lint`
- `npm run galaxy-maps` ‚Äî rebakes the Milky Way detail/volume textures and the Local Group portraits from `artwork/` sources. Only needed after touching `app/galaxyPaint.ts` or the authored art.
- `npm run universe-field` ‚Äî rebakes the committed cosmic-web density gzip from `app/universeField.ts`. Only needed after touching that generator.
- `npm run universe-cmb` ‚Äî rebakes the CMB boundary texture from the committed WMAP source. Only needed after touching `scripts/bake-universe-cmb.mjs` or the source image.
- `npm run narration` ‚Äî renders any new/changed `NARRATION` lines via Kokoro (`scripts/render-narration.mjs`) and updates the cache in `.narration-cache/`. Requires the local Kokoro venv (see that script's header).

Single test: `node --test --experimental-strip-types tests/orbits.test.mjs` (no build needed). The export test needs an existing `out/`. The `--experimental-strip-types` flag is what lets the `.mjs` tests import the `.ts` modules directly.

## Architecture

**Rendering path.** `app/page.tsx` ‚Üí `app/ClientAtlas.tsx` ‚Üí `app/CosmicAtlas.tsx` ‚Üí `app/SolarSystem.tsx`, `app/DeepSpace.tsx` (Milky Way and Local Group ‚Äî one component, `mode` prop selects which), or `app/Universe.tsx` (the observable universe). `ClientAtlas` loads the atlas via `next/dynamic` with `ssr: false` because the scenes need DOM/WebGL at import time. The prerendered HTML is just a shell.

Core modules:

- `app/orbits.ts` ‚Äî the two-body math. **No Three.js, no DOM**, so `node --test` can exercise it directly. Keep it that way; it is the one part of the app where a wrong answer looks exactly like a right one on screen.
- `app/bodies.ts` ‚Äî the body catalog (physical data + JPL elements) and `NARRATION` (the kid-facing lines spoken by Ambient mode and by the tvOS tour, keyed by body name plus the three finale keys `"Milky Way"`, `"Local Group"`, `"Universe"`). Also Three.js-free, so the tests check the *real* elements rather than a copy that would drift.
- `app/cosmic.ts` ‚Äî typed Milky Way landmarks (`GALACTIC_REGIONS`), Local Group galaxies (`NEARBY_GALAXIES`), Observable Universe landmarks (`UNIVERSE_LANDMARKS`), and the seven-stop `COSMIC_JOURNEY` that walks all four scales in order, ending at the CMB. This is the source of truth for educational deep-space content.
- `app/SolarSystem.tsx` ‚Äî the Three.js scene and all UI (~640 lines) for the innermost scale. `heliocentricVector` / `orbitVectors` are the only bridge from the math to `THREE.Vector3`.
- `app/DeepSpace.tsx` ‚Äî the Milky Way / Local Group Three.js renderer (one component, two normalized coordinate spaces, neither tied to the solar-system simulation).
- `app/Universe.tsx` ‚Äî the observable-universe renderer: the raymarched cosmic web, the deep-field galaxy point layer, the CMB boundary sphere, and the eleven landmark markers.
- `app/deepOverlays.ts` ‚Äî the marker/label/occlusion/raycast screen-space plumbing shared by `DeepSpace.tsx` and `Universe.tsx` (extracted verbatim from `DeepSpace.tsx`, zero behavior change, so both deep-space renderers speak the same overlay language without copy-pasting it).
- `app/CosmicAtlas.tsx` ‚Äî owns scale switching, cinematic transitions, the Cosmic Address journey, and (see Ambient mode below) the lifted Ambient orchestration that has to survive renderer unmounts. It unmounts one renderer before mounting the next so multiple animation loops never run at once.

**The Milky Way and Observable Universe volumes.** Two raymarched glows, built on different math because they're different shapes:

- `app/galaxyPaint.ts` ‚Äî Three-free, DOM-optional deterministic galaxy/Andromeda/Local-Group-portrait painter (canvases in, canvases out via an injectable `CanvasFactory`) so `scripts/bake-galaxy-maps.mjs` can run the exact same painter under Node with `@napi-rs/canvas` ‚Äî one brush, two runtimes, zero drift between what the web paints live and what gets baked for the authored-detail overlay and the tvOS bundle.
- `app/galaxyVolume.ts` / `tvos/HeliosTV/GalaxyVolume.metal` ‚Äî the Milky Way's raymarcher. The galaxy is a **60:1 pancake**, so this is deliberately a **2.5D trick**: a 2D emission/dust texture pair extruded through an analytic vertical density profile, not a 3D volume texture ‚Äî cheap, and exact for a genuinely flat structure.
- `app/galaxyDetail.ts` ‚Äî loads the baked authored-detail surface and picks a quality tier (`chooseGalaxyDetailQuality`) from GPU/memory/pointer signals.
- `app/localGroupAssets.ts` ‚Äî loads the authored Milky Way/Andromeda/Triangulum portraits for the Local Group scale, one shared texture set with the tvOS bundle.
- `app/universeField.ts` ‚Äî Three-free deterministic 128¬≥ RGBA density field generator (Worley-ridge cosmic web statistics + named-structure stamps for Laniakea/Shapley/the Bo√∂tes Void/the Sloan Great Wall, with an anchor-mask alpha channel marking which voxels are catalog-charted vs. purely statistical). `scripts/bake-universe-field.mjs` bakes this to a committed gzip ‚Äî the field is genuinely 3D, so unlike the galaxy's 2.5D trick there is no cheaper representation, and baking it avoids ever generating 2M+ voxels on the main thread.
- `app/universeVolume.ts` / `tvos/HeliosTV/UniverseVolume.metal` ‚Äî the cosmic-web raymarcher (a real `Data3DTexture`/`texture3d` march, segment-scaled step count, interleaved-gradient-noise jitter) plus `createUniverseGalaxies`, a deterministic ~40k-point deep-field galaxy layer importance-sampled from the same field so resolved points sit exactly on the filaments the volume glows along. Both renderers ship this same one-two: **volume for the aggregate glow, points for individual sparkle** ‚Äî the identical fix that first shipped on the Milky Way scale (see git history: "Sharpen the Milky Way: pixel stars, tighter slices, gentler bloom") transposed up one scale. Camera-inside-the-volume must always work (the finale flies through it); the shader's `tNear = max(tNear, 0)` clamp is what makes that safe.
- `scripts/bake-universe-cmb.mjs` ‚Äî bakes the CMB boundary texture from the committed WMAP source (`artwork/universe/`, public domain, credit NASA/WMAP Science Team ‚Äî ESA's Planck map was rejected for its Standard Licence). It does not just reproject: it **recovers real per-pixel temperature from the source colormap's hue** (empirically monotonic 0¬∞‚Üí259.09¬∞ across this exact file) and re-renders it through the volume's own cool/warm duotone, gamma-biased toward a near-background baseline so only genuine extremes brighten ‚Äî a straight recolor of the raw rainbow paints roughly half the sky solid orange, because the real signal is ~zero-centered. If you ever swap the source image, re-derive that hue range; it is not a generic constant.

**Perf lessons that apply to any raymarch here** (measured on real Apple TV hardware, not simulator ‚Äî see git history for the `[perf]` traces): step count should scale with the segment actually marched, not be a fixed ceiling; resolution/render-scale reduction is worth far more than step-count trims because these shaders are fill-bound; and a hidden `SCNProgram` node's first draw compiles its Metal pipeline and uploads its textures inline ‚Äî pre-warm via `SCNSceneRenderer.prepare()` before you need it, or eat a 300ms+ hitch the frame it unhides.

**The imperative bridge (most important invariant).** The entire Three.js scene is constructed in one mount-once `useEffect` with an empty dependency array. React state does *not* drive the scene through re-renders. Instead the effect publishes `apiRef.current = { focus, scale, date, previewDate }`, and UI handlers call those methods. Adding a dependency to that effect would tear down and rebuild the whole scene on every state change. Keep new scene mutations inside the effect and expose them through `apiRef`.

**Position math.** Planets carry `elements` (JPL approximate Keplerian elements, `base` + `rate` per Julian century since J2000); everything else carries `smallBody` (JPL SBDB elements with an epoch and mean motion). Both flow through `orbitParamsAt` ‚Üí `solveKepler` ‚Üí `heliocentricPosition`.

`solveKepler` uses a **Danby starter plus Halley iteration**, not Newton-from-`E = M`. This is not incidental. Newton-from-M diverges as eccentricity approaches 1 ‚Äî the derivative `1 - e¬∑cos E` collapses near perihelion and it converges on the *wrong root* without ever erroring. It put NEOWISE (e=0.9992) and Hale‚ÄìBopp (e=0.9950) hundreds of AU off their true positions for roughly a fifth of the supported date range, silently, for as long as the feature existed. `tests/orbits.test.mjs` pins every body to a bracketed reference solver across all of 1800‚Äì2050. **If you touch the solver, run those tests** ‚Äî the failure mode is invisible on screen.

These are **two-body approximations** ‚Äî they ignore perturbations, so accuracy degrades away from each body's epoch (Pluto's elements are from 2016 and Neptune tugs on it). That is a deliberate, accepted tradeoff: this is a teaching tool. The UI says so in the footer; keep that framing honest if you change the math.

**Asset paths and `basePath`.** GitHub Pages serves this project repo from `/solar-system-atlas`, not the domain root. Next rewrites its own asset URLs for `basePath`, but **Three.js loads textures from raw strings, which Next cannot rewrite** ‚Äî so `ASSET_BASE` (from `NEXT_PUBLIC_BASE_PATH`, set in the deploy workflow) is prefixed onto texture URLs manually in `SolarSystem.tsx`. The favicon in `layout.tsx` is prefixed the same way. Get this wrong and textures 404 *silently*: planets render as flat grey spheres with no console error. Always verify texture loads in a browser after touching asset paths.

**Dates are UTC, everywhere.** Positions are computed at UTC noon and the panels format with `timeZone:"UTC"`, so every calendar-date string in the app comes from `dateValue` (UTC). Deriving any of them from local time reintroduces a day-off mismatch between the date chip and the SIMULATION DATE label. `MIN_DATE_VALUE` / `MAX_DATE_VALUE` are the single source for the supported range: the `<input type="date">` bounds, the playback clamp, and `clampDateValue` all derive from them. Note that `min`/`max` on a date input only *marks* a typed value invalid ‚Äî it still fires `change` ‚Äî so manual entry must go through `clampDateValue`.

**Scale modes.** `distanceScale(au, mode)` is the single source of truth: `readable` is `sqrt(au) * 29` (compressed distances, real orbital angles), `linear` is `au * 5.5` (true relative distance, planets enlarged ~0.36√ó, Sun 0.08√ó). It feeds orbit lines, labels, belt point clouds, and the invisible torus click targets for the asteroid/Kuiper belts ‚Äî changing it means updating all of those together (see `rebuildScale` and `updateRegions`).

**Ambient mode** (`app/ambient.ts`, `app/useAmbient.ts`) is a lean-back, narrated auto-tour ‚Äî a third mode beside Explore and Guided Tour, and the web sibling of the tvOS app. It flies outward Sun-to-Eris speaking one fact per world, then ‚Äî matching the tvOS finale ‚Äî recedes through three more narrated beats (Milky Way, Local Group, cosmic web) before returning to the Sun and looping. Five things are worth knowing. (1) The orchestration (the `useAmbient` hook: drone, caption, step sequencing) lives in `CosmicAtlas.tsx`, not inside `SolarSystem.tsx` ‚Äî it has to survive `CosmicAtlas` swapping `SolarSystem` for `DeepSpace`/`Universe` on every scale change, so anything tied to `SolarSystem`'s own lifecycle would die the moment the finale tried to leave it. `SolarSystem` receives its scene `apiRef` and the hook's return value (`AmbientState`) as props instead of creating either locally; its own mount-once effect is untouched. (2) During the Sun‚ÜíEris leg, the camera is driven through `flyTo(name, onArrive)` and `setAmbient(on)` on that `apiRef`, so the mount-once scene effect is never rebuilt. During the three finale beats there is no equivalent arrival callback in `DeepSpace.tsx`/`Universe.tsx` ‚Äî the orchestrator calls `CosmicAtlas`'s existing `navigate(mode)` and waits out its own transition timing, then lets each scene settle at its default wide framing rather than focusing a specific landmark. (3) Audio needs a user gesture (browser autoplay policy), which is why there is a "Begin" gate; the Web Audio drone (`createDrone`, ported from the tvOS `AmbientAudio.swift`) and the narration clips both start from that click, and the SAME drone instance keeps running across all four scales/finale beats since it lives in `CosmicAtlas`. (4) Leaving ambient mode ‚Äî the exit button or Escape, from anywhere including mid-finale ‚Äî always navigates back to the Solar System; every async step in the finale sequence re-checks the existing cancellation token so a mid-flight exit can never resurrect a stale step or strand the drone. (5) The 183 narration clips in `public/narration/` (including the three finale beats) are generated by Kokoro (`npm run narration`) but **committed**, because GitHub Pages cannot run the renderer at deploy time ‚Äî same reasoning as any built asset that the CI environment cannot reproduce.

**Moons** are display-only approximations. `moonOrbitRadius` is deliberately *logarithmic*, not physical, so moons stay visible next to their parent; orbits are circular from a stored `phase` rather than solved. Moons are hidden unless their family is selected (`showMoonFamily`). The UI labels this honestly ("DISPLAY SPACING COMPRESSED") ‚Äî keep that framing.

**WebGL fallback.** If `new THREE.WebGLRenderer()` throws, the mount gets a `.no-webgl` class and a pure-CSS orbit diagram (`.fallback-*` in `globals.css`) takes over while panels and the tour stay interactive. Note that most headless browsers cannot create a WebGL context and will hit this path ‚Äî to QA the real scene you need a headed browser with GPU access.

**Galactic scales.** Do not put the Milky Way, Local Group, or Observable Universe into the solar-system coordinate system, or into each other's. The scale jumps are too large to remain navigable or numerically meaningful ‚Äî each is its own independent normalized space with its own home marker, and the "you are here" continuity across scales is carried by narration and marker placement, not shared coordinates. `DeepSpace` covers the first two: the Milky Way is a reconstructed particle-plus-volume visualization with the Sun at roughly 26,000 light-years from the center; Local Group radial distances are proportional to catalog distances from the Milky Way, but galaxy diameters are enlarged and directions are schematic. `Universe.tsx` is the fourth: the cosmic-web field is a **statistical illustration** past roughly a gigalight-year (real named structures ‚Äî Laniakea, the Sloan Great Wall, and the rest ‚Äî are stamped into it at compressed schematic positions; the honesty copy calls objects beyond that radius, like 3C 273 and JADES-GS-z14-0, explicit "schematic pointers"), and the CMB boundary sphere is the observable universe's genuine edge ‚Äî the oldest light there is, enclosing everything, in every direction, for real. Keep those qualifications visible in the UI; they are the whole reason the app is trustworthy.

**Textures.** `TEXTURE_MAPS` maps bodies to real NASA-derived images in `public/textures/` (WebP, 2048√ó1024, ~2.4 MB total; they all load eagerly and gate the loading veil); anything absent falls back to `makePlanetTexture`, a procedural canvas gradient. The info panel's "VISUAL MAP" line surfaces which one is in use. Attribution requirements (CC BY 4.0 / NASA credit) are in `public/textures/README.md` ‚Äî maintain them, and keep the "approximation" wording for Haumea, Makemake, and Eris, which have no resolved surface maps. `public/textures/galaxy/`, `public/textures/local-group/`, and `public/textures/universe/` hold the baked deep-space assets described above; unlike the planet textures these are all generated (or, for the CMB, recolored) by the `scripts/bake-*` family rather than downloaded as-is, and `public/textures/README.md` documents the exact source and license for each.

**Styling** is hand-written CSS in `app/globals.css` (dense, one-rule-per-line). Tailwind is imported but effectively unused ‚Äî follow the existing CSS rather than introducing utility classes.

## Conventions

- `SolarSystem.tsx`, `DeepSpace.tsx`, `Universe.tsx`, and the generator/volume modules they depend on are all written in the same deliberately compact style: packed one-liners, no spaces around operators, minimal comments reserved for non-obvious constraints. Match it rather than reformatting.
- The 1800‚Äì2050 date bound is enforced in two places that must stay in sync: the `<input type="date">` min/max, and `MIN_SIM_TIME` / `MAX_SIM_TIME` for playback.
- Static export means no server-side anything: no route handlers, no `headers()`/`cookies()`, no `next/image` optimization (`images.unoptimized` is set). Adding a feature that needs a server means leaving GitHub Pages.
